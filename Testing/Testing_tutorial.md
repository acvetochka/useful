## Ручне та автоматизоване тестування

`Ручне тестування`: Під час ручного тестування тестер вручну переглядає кожну частину програми та тестує різні функції, щоб переконатися, що вона працює правильно. Якщо в процесі тестування виявляються будь-які помилки, тестер повідомляє про них команді розробників програмного забезпечення, щоб їх можна було виправити.
`Автоматизоване тестування`: В автоматизованому тестуванні ви можете автоматизувати свої тести, написавши окрему програму, яка перевіряє, чи поводиться ваша програма належним чином.

## Модульне тестування
`Модульне тестування`: Під час модульного тестування ви перевіряєте кожну функцію, щоб переконатися, що все працює належним чином. Модульні тести також можуть служити формою документації для вашої програми, оскільки вони призначені для представлення очікуваної поведінки вашого коду.

`Принцип єдиної відповідальності`: Принцип єдиної відповідальності рекомендує, щоб кожна функція була невеликою та відповідала за одну річ.

`Поширені фреймворки для тестування JavaScript`: Деякі поширені фреймворки для тестування включають Jest, Mocha та Vitest. Jest — це популярний фреймворк для модульних тестів.
Ось приклад модульних тестів з використанням Jest.

Спочатку ви можете створити функцію, яка відповідає за повернення щойно відформатованого рядка:
```js
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

В окремому  `getFormattedWord.test.js` файлі ви можете написати кілька тестів, щоб перевірити, чи функція працює належним чином. Файл  getFormattedWord.test.js виглядатиме так:
```js
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

`expect` Функція : Функція  expect використовується для перевірки значення.
`Matcher` : Matcher — це функція, яка перевіряє, чи значення поводиться належним чином. У наведеному вище прикладі matcher —  toBe(). Jest має різноманітні matcher-и.
Щоб використовувати Jest, спочатку потрібно встановити  jest пакет за допомогою  npm i jest. Також потрібно буде додати скрипт до вашого  package.json файлу, ось такий:
```json
"scripts": {
  "test": "jest"
},
```
Потім ви можете виконати команду npm run test для запуску тестів.

## Життєвий цикл розробки програмного забезпечення

### Різні етапи життєвого циклу розробки програмного забезпечення :

- `Етап планування` : Команда розробників збирає вимоги до запропонованої роботи від зацікавлених сторін.
- `Етап проектування` : Команда розробників програмного забезпечення аналізує вимоги та визначає найкращі підходи до рішень.
- `Етап впровадження` : Команда розробників програмного забезпечення розбиває вимоги на керовані завдання та реалізує їх.
- `Етап тестування` : Він включає ручне та автоматизоване тестування нової роботи. Іноді команда тестує застосунок протягом усього етапу розробки, щоб виявити та виправити будь-які проблеми, що виникають.
- `Етап розгортання` : Команда розгортає нові зміни в середовищі збірки або тестування.
- `Етап обслуговування` : це включає виправлення будь-яких проблем, що виникають у клієнтів у виробничому застосунку.

### Різні моделі життєвого циклу розробки програмного забезпечення :

- `Модель водоспаду` : Модель водоспаду полягає в тому, що кожен етап життєвого циклу має бути завершений до початку наступного етапу.
- `Гнучка модель` : Гнучка модель зосереджена на ітеративній розробці, розбиваючи роботу на спринти.

## БДД та ТДД
- `TDD` : Розробка на основі тестування (TDD) – це методологія, яка наголошує на написанні тестів спочатку. Написання тестів перед створенням функцій забезпечує зворотний зв'язок у режимі реального часу для розробників під час процесу розробки.
- `BDD` : Розробка, орієнтована на поведінку, – це підхід, що полягає у узгодженні серії тестів з бізнес-цілями. Тестові сценарії в BDD повинні бути написані мовою, зрозумілою як для технічних, так і для нетехнічних фахівців. Прикладом такого синтаксису є Gherkin.
- `Фреймворки для тестування BDD` : Прикладами фреймворків для тестування BDD є Cucumber, JBehave та SpecFlow.

## Твердження в модульному тестуванні
`Твердження` : Твердження використовуються для перевірки того, чи код поводиться належним чином.
`Бібліотеки тверджень` : Chai — це поширена бібліотека тверджень. Інші поширені бібліотеки тверджень JavaScript —  should.js це  expect.js.
Ось приклад твердження з використанням Chai, яке перевіряє, чи повернене значення з  addThreeAndFour функції дорівнює числу 7:
```
assert.equal(addThreeAndFour(), 7);
```

### Найкращі практики : 
Незалежно від того, яку бібліотеку тверджень ви використовуєте, вам слід писати чіткі повідомлення про твердження та невдачі, які допоможуть вам зрозуміти, які тести не проходять і чому.

Глузування, прикидання та знущання
- `Імітація` : Імітація — це процес заміни реальних даних хибними даними, що імітує поведінку реальних компонентів. Наприклад, ви можете імітувати відповідь API під час тестування замість того, щоб здійснювати безперервні виклики API для отримання даних.
- `Заглушки` : Заглушки – це об'єкти, які повертають попередньо визначені відповіді або фіктивні дані для очікуваної поведінки в застосунку. Наприклад, ви можете заглушити поведінку для підключення до бази даних у своїх тестах, не покладаючись на фактичне підключення до бази даних.
- `Підробка` : Підробки – це спрощені версії реальних компонентів без складності чи побічних ефектів. Наприклад, ви можете підробити базу даних, зберігаючи дані в пам'яті, замість того, щоб взаємодіяти зі справжньою базою даних. Це дозволить вам імітувати операції з базою даних у пам'яті, що буде набагато швидше, ніж робота зі справжньою базою даних.

## Функціональне тестування
`Функціональне тестування` : Функціональне тестування перевіряє, чи працюють функції та можливості програми належним чином. Метою функціонального тестування є тестування системи в цілому за кількома сценаріями.

`Нефункціональне тестування` : Нефункціональне тестування зосереджується на таких речах, як продуктивність та надійність.

`Димове тестування`: Димове тестування – це попередня перевірка системи на наявність основних або критичних проблем перед початком більш масштабного тестування.

## Наскрізне тестування
`Наскрізне тестування` : Наскрізне тестування, або E2E, перевіряє реальні сценарії з точки зору користувача. Наскрізні тести допомагають забезпечити правильну роботу вашої програми та її передбачуваність для користувачів. Однак їх налаштування, проектування та обслуговування займають багато часу.

- `Фреймворки для комплексного тестування` : Playwright — це популярний фреймворк для комплексного тестування, розроблений Microsoft. Інші приклади інструментів комплексного тестування включають Cypress, Selenium та Puppeteer.
Ось приклад E2E-тестів з кодової бази freeCodeCamp з використанням Playwright.  beforeEach Хук буде виконуватися перед кожним тестом. Тести перевіряють, чи має донор посилання для підтримки в рядку меню, а також спеціальну стилізовану рамку навколо свого аватара:
```
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## Тестування зручності використання

`Тестування зручності використання` : Тестування зручності використання – це коли реальні користувачі взаємодіють із застосунком, щоб виявити, чи є в ньому якісь проблеми з дизайном, користувацьким досвідом або функціональністю. Тестування зручності використання зосереджується на інтуїтивно зрозумілій функціональності застосунку для користувачів.

**Чотири поширені типи тестування зручності використання **:
- `Дослідницьке` : Дослідницьке тестування зручності використання передбачає взаємодію користувачів з різними функціями програми, щоб краще зрозуміти, як вони працюють.
- `Порівняльне` : Порівняльне тестування – це порівняння взаємодії користувача з вашим застосунком з аналогічними застосунками на ринку.
- `Оцінювання` : Оціночне тестування – це те, під час якого ви вивчаєте, наскільки інтуїтивно зрозумілим є використання програми.
- `Валідація` : Валідаційне тестування – це процес виявлення будь-яких серйозних проблем, які заважатимуть користувачеві ефективно використовувати програму.

Інструменти для тестування зручності використання : Прикладами інструментів для тестування зручності використання є Loop11, Maze, Userbrain, UserTesting та UXTweak.

## Тестування сумісності
`Тестування сумісності` : Метою тестування сумісності є забезпечення роботи вашої програми в різних обчислювальних середовищах.

### Різні типи тестування на сумісність :
- `Зворотна сумісність` : Зворотна сумісність означає, що програмне забезпечення сумісне з попередніми версіями.
- `Сумісність уперед` : Сумісність уперед стосується того, коли програмне забезпечення та системи працюватимуть з майбутніми версіями.
- `Сумісність обладнання` : Сумісність обладнання – це здатність програмного забезпечення належним чином працювати в різних конфігураціях обладнання.
- `Сумісність операційних систем`: Сумісність операційних систем – це здатність програмного забезпечення працювати на різних операційних системах, таких як macOS, Windows та дистрибутиви Linux, такі як Ubuntu та Fedora.
- `Сумісність із мережею` : Сумісність із мережею означає, що програмне забезпечення може працювати в різних мережевих умовах, таких як різні швидкості мережі, протоколи, налаштування безпеки тощо.
- `Сумісність браузерів` : Сумісність браузерів означає, що веб-застосунок може працювати однаково в різних браузерах, таких як Google Chrome, Safari, Firefox тощо.
- `Сумісність з мобільними пристроями` : важливо переконатися, що ваші програмні застосунки працюють на різних пристроях Android та iOS, включаючи телефони та планшети.

## Тестування продуктивності

`Тестування продуктивності` : Під час тестування продуктивності ви перевіряєте швидкість роботи, швидкість реагування, масштабованість та стабільність програми за різних робочих навантажень. Мета полягає у вирішенні будь-яких типів вузьких місць у продуктивності.

### Різні типи тестування продуктивності :
- `Тестування навантаження` : Тестування навантаження визначає, як система поводиться під час нормального та пікового навантаження.
- `Стрес-тестування` : Стрес-тестування – це тестування вашої програми в умовах екстремальних навантажень та перевірка того, наскільки добре ваша система реагує на підвищене навантаження.
- `Тестування замочуванням` (випробування на витривалість) : Випробування замочуванням або витривалість – це тип навантажувального тестування, під час якого система тестується з більшим навантаженням протягом тривалого періоду часу.
- `Тестування пікових навантажень` : Тестування пікових навантажень – це процес різкого збільшення та зменшення навантажень, а також аналізу реакції системи на ці зміни.
- `Тестування точок зупинки` (тестування ємності) : Тестування точок зупинки або тестування ємності – це повільне збільшення навантаження з часом до точки, коли система починає давати збої або деградувати.

## Тестування безпеки
`Тестування безпеки `: Тестування безпеки допомагає виявити вразливості та слабкі місця.

### Принципи безпеки :
- `Конфіденційність` : це захищає від розголошення конфіденційної інформації іншим одержувачам, які не є цільовим одержувачем.
- `Цілісність` : це передбачає запобігання зміні інформації користувача зловмисними користувачами.
- `Автентифікація` : це передбачає перевірку особи користувача, щоб переконатися, що йому дозволено використовувати цю систему.
- `Авторизація` : це процес визначення того, які дії дозволено виконувати автентифікованим користувачам або до яких частин системи їм дозволено доступ.
- `Доступність` : це гарантує, що інформація та послуги доступні авторизованим користувачам, коли вони їм потрібні.
- `Незаперечність` : це гарантує, що як відправник, так і одержувач мають підтвердження доставки та підтвердження особи відправника. Це захищає від заперечення відправником факту надсилання інформації.

### Поширені загрози безпеці :
- `Міжсайтовий скриптинг (XSS)` : XSS-атаки відбуваються, коли зловмисник впроваджує шкідливі скрипти на веб-сторінку, а потім виконує їх у контексті браузера жертви
- `SQL-ін'єкція` : SQL-ін'єкція дозволяє зловмисникам впроваджувати шкідливий код у базу даних.
- `Атака типу «відмова в обслуговуванні» (DoS)` : DoS-атака відбувається, коли зловмисники завалюють веб-сайт великою кількістю запитів або трафіку, що призводить до уповільнення роботи сервера та, можливо, до його збою, що робить сайт недоступним для користувачів.

### Категорії інструментів тестування безпеки :
- `Статичне тестування безпеки застосунків`: ці інструменти оцінюють вихідний код застосунку для виявлення вразливостей безпеки.
- `Динамічне тестування безпеки застосунків` : ці інструменти взаємодіють з інтерфейсом застосунку для виявлення потенційних вразливостей безпеки. Інструменти DAST не мають доступу до вихідного коду.
- `Тестування на проникнення (пентест)` : Тестування на проникнення – це тип тестування безпеки, який передбачає створення симульованих кібератак на додаток для виявлення будь-яких вразливостей у системі.

## A/B-тестування
- `A/B-тестування` : A/B-тестування передбачає порівняння двох версій сторінки або програми та вивчення того, яка версія працює краще. Воно також відоме як сегментне або спліт-тестування. A/B-тестування дозволяє приймати більше рішень на основі даних та постійно покращувати користувацький досвід.
- `Інструменти для A/B-тестування` : Прикладами інструментів для A/B-тестування є GrowthBook та LaunchDarkly.

## Альфа- та бета-тестування
Після завершення початкової розробки та тестування програмного забезпечення важливо, щоб застосунок протестували тестувальники та реальні користувачі. Саме тут на допомогу приходять альфа- та бета-тестування.

- `Альфа-тестування` : Альфа-тестування проводиться відібраною групою тестувальників, які перевіряють додаток, щоб переконатися у відсутності помилок перед його випуском на ринок. Альфа-тестування є частиною приймального тестування та використовує як методи білого, так і чорного ящика.
- `Бета-тестування` : Бета-тестування – це процес, під час якого програма стає доступною для реальних користувачів. Користувачі можуть взаємодіяти з програмою та залишати відгуки. Бета-тестування також є формою тестування прийняття користувачами.
- `Приймально-приймальні випробування` : Приймально-приймальні випробування гарантують, що програмний застосунок відповідає бізнес-вимогам та потребам користувачів перед його випуском.
- `Тестування чорної скриньки` : Тестування чорної скриньки зосереджується лише на очікуваній поведінці програми.
- `Тестування білої скриньки` : Тестування білої скриньки передбачає знання тестувальником внутрішніх компонентів та проведення на них тестів.

## Регресійне тестування
- `Регресія` : Регресія стосується ситуацій, коли нові зміни ненавмисно порушують існуючу функціональність.
- `Регресійне тестування` : Регресійне тестування допомагає виявити проблеми регресії. Під час регресійного тестування ви повторно запускаєте функціональні тести для частин вашої програми, щоб переконатися, що все працює належним чином.

### Інструменти для регресійного тестування : інструменти, які можна використовувати для виконання регресійного тестування, включають Puppeteer, Playwright, Selenium та Cypress.
### Методи регресійного тестування :
- `Модульне регресійне тестування` : тут ви знайдете список елементів, які потрібно тестувати щоразу, коли в додаток вносяться значні зміни або виправлення.
- `Часткове регресійне тестування` : це включає цілеспрямовані підходи, щоб переконатися, що нові зміни не порушили певні аспекти програми.
- `Повне регресійне тестування` : це тестування всіх функцій у кодовій базі. Це найбільш трудомісткий та детальний варіант.
- `Повторне тестування` : Повторне тестування використовується для перевірки відомих проблем та переконання в тому, що вони вирішені. На відміну від цього, регресійне тестування шукає невідомі проблеми, які могли виникнути внаслідок нещодавніх змін у кодовій базі.
