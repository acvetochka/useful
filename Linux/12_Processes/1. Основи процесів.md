## 1. Що таке процес
**Просте визначення**

`Процес` — це запущена програма, яка зараз виконується або готова до виконання.

Наприклад:
- ти запускаєш bash
- відкриваєш python
- стартуєш node app.js
- запускаєш nginx

Усі ці запущені екземпляри — це процеси.

**Більш точне визначення**

`Процес` — це не просто “програма в роботі”, а окремий об’єкт, яким керує ядро Linux.

Ядро для кожного процесу зберігає інформацію:
- хто його запустив
- який у нього PID
- скільки пам’яті він використовує
- які файли відкриті
- який у нього поточний стан
- які права доступу він має
- скільки CPU часу він уже отримав
- хто його батько
- де лежить його код, стек, купа в пам’яті
- які сигнали він отримав або ігнорує

Тобто процес — це не лише код, а цілий набір даних + стан виконання.

**Інтуїтивно**

Можна уявити так:
- програма — це рецепт
- процес — це вже приготування страви за цим рецептом

Один і той самий рецепт можна використати багато разів.
Так само одну й ту саму програму можна запустити багато разів, і кожен запуск буде окремим процесом.

Наприклад, якщо відкрити 3 термінали, то буде 3 окремих процеси `shell`, навіть якщо всі вони запускають один і той самий `/bin/bash`.

## 2. Програма vs процес

Це одна з найважливіших відмінностей у Linux.

**Програма**

`Програма` — це файл з інструкціями, який лежить на диску.

Наприклад:
- /bin/ls
- /usr/bin/python3
- /usr/bin/vim
- /usr/sbin/nginx

Програма сама по собі:
-  не виконується
- не має PID
- не займає оперативну пам’ять як активний об’єкт виконання
- не має поточного стану running/sleeping/zombie

Програма — це просто пасивний файл.

**Процес**

`Процес` — це вже активне виконання програми.

Коли ти запускаєш програму, ядро:
- створює новий процес
- виділяє йому ресурси
- завантажує код у пам’ять
- налаштовує стек, heap, змінні оточення
- дає процесу PID
- ставить його в чергу планувальника

І лише після цього програма стає процесом.

Один файл — багато процесів

Одна програма може породити багато процесів.

Приклад:
```bash
sleep 100
sleep 100
sleep 100
```

> 👉 тут:
> - sleep — це програма  
> - 100 — це кількість секунд, які вона повинна "спати"  

Це можуть бути 3 окремі процеси sleep, кожен зі своїм PID.
***

**Один процес — не обов’язково назавжди одна й та сама програма**

Це вже тонший момент.

У Linux процес може замінити свій код і дані через сімейство системних викликів exec().

Тобто:
1. shell створює процес
2. той процес викликає exec
3. замість старої програми в ньому запускається нова

PID при цьому може залишитися той самий, але в процесі вже виконується інша програма.

Це важливо, бо показує:
> процес — це не просто “назва команди”, а сутність, якою керує ядро.

## 3. Що входить до складу процесу

Коли кажуть “процес”, мають на увазі не лише виконуваний код. Процес включає багато компонентів.

### 3.1 Код програми (text segment)

Це машинні інструкції, які CPU виконує.

Наприклад:
- інструкції python
- інструкції bash
- інструкції твоєї compiled-програми на C/C++

### 3.2 Дані процесу

Це:
- глобальні змінні
- статичні змінні
- дані, завантажені під час роботи

### 3.3 Стек (stack)

`Стек` використовується для:
- локальних змінних функцій
- адрес повернення з функцій
- параметрів виклику
- службової інформації викликів

Коли функція викликається — стек росте.  
Коли функція завершується — частина стеку звільняється.

### 3.4 Купа (heap)

`Heap` — це область пам’яті для динамічного виділення пам’яті.

Наприклад:
- malloc() у C
- new у C++
- динамічні об’єкти в багатьох мовах

### 3.5 Відкриті файли

У Linux майже все є файлом або файловоподібним об’єктом.

Процес може мати відкритими:
- звичайні файли
- каталоги
- сокети
- пайпи
- пристрої
- stdin, stdout, stderr

Ядро веде таблицю відкритих файлових дескрипторів процесу.

### 3.6 Атрибути безпеки процесу (Security Attributes)

Кожен процес у Linux має набір атрибутів, які визначають:

- хто він
- від чийого імені працює
- що йому дозволено робити

Це основа всієї моделі безпеки Linux.

- **1. real UID (RUID — реальний UID)**

  👉 Це ідентифікатор користувача, який запустив процес
  - встановлюється при створенні процесу
  - зазвичай відповідає користувачу в системі (/etc/passwd)

  📌 Приклад

  Якщо користувач `user` запускає:
  ```bash
  id="x1"
  ls
  ```
  то:
  > RUID = UID користувача `user`

- **2. effective UID (EUID — ефективний UID)**

  👉 Це ідентифікатор, від якого процес реально виконує дії

  💥 Найважливіший атрибут безпеки

  Саме він визначає:
  - чи може процес читати файл
  - чи може записувати
  - чи може відкривати сокети
  - чи має root-права

  📌 Приклад з sudo
  sudo nano /etc/passwd

  RUID = твій користувач

  EUID = root (0)

  👉 тому процес має права root

  📌 setuid

  Якщо файл має біт setuid, то:

  > процес отримує EUID власника файлу

  Наприклад:
  ```bash
  ls -l /usr/bin/passwd
  ```
  часто має:
  ```
  -rwsr-xr-x
  ```
  👉 це означає:
  > запускається від імені root (EUID = 0)

- **3. saved UID (SUID)**

  👉 Менш очевидний, але важливий
  - зберігає попередній EUID
  - дозволяє процесу тимчасово змінювати привілеї і повертати їх назад

- **4. real GID (RGID)**

  👉 Ідентифікатор групи користувача, який запустив процес

- **5. effective GID (EGID)**

  👉 Група, від якої процес реально виконує операції

  Аналогічно до UID:
  - RGID — хто запустив
  - EGID — від чиєї групи працює

- **6. supplementary groups (додаткові групи)**

  👉 Список усіх груп, до яких належить користувач

  Наприклад:
  ```bash
  id
  ```
  ```bash
  uid=1000(user) gid=1000(user) groups=1000(user),4(adm),24(cdrom)
  ```
  👉 процес успадковує ці групи

  📌 Навіщо це потрібно

  Це дозволяє:
  - доступ до різних ресурсів
  - розмежування прав
  - роботу з файлами різних груп

-  **7. capabilities (Linux capabilities)**

    👉 Це сучасна альтернатива “все або нічого” для root

    Замість:

    ❌ або root (все дозволено)  
    ❌ або user (майже нічого)

    Linux дозволяє:

    👉 дати процесу конкретні привілеї

    📌 Приклади capabilities
    | Capability           | Що дозволяє                      |
    | -------------------- | -------------------------------- |
    | CAP_NET_BIND_SERVICE | відкривати порти < 1024          |
    | CAP_SYS_TIME         | змінювати системний час          |
    | CAP_SYS_ADMIN        | майже root                       |
    | CAP_KILL             | надсилати сигнали іншим процесам |
    📌 Приклад

    Дати програмі право відкривати порт 80 без root:
    ```bash
    sudo setcap 'cap_net_bind_service=+ep' ./server
    ```
🔥 Висновок

👉 Доступ процесу визначається НЕ лише користувачем, а комбінацією:

- UID / GID
- effective UID / GID
- supplementary groups
- capabilities

### 3.7 Стани процесу (Process States)

Процес у Linux не просто “працює або ні”, а має різні стани.

-  **1. running (виконується)**

    👉 процес прямо зараз виконується на CPU
    - отримав квант часу
    - виконує інструкції

- **2. runnable (готовий до виконання)**

  👉 процес готовий працювати, але чекає CPU
  - стоїть у черзі scheduler
  - не виконується прямо зараз

  💡 важливо:  
    у top часто це теж позначається як `R`

- **3. sleeping (сплячий)**

  👉 процес чекає на подію

  Наприклад:
  - введення з клавіатури
  - завершення I/O
  - відповідь від мережі
  - таймер

  Є два типи:  
  🟢 interruptible sleep (S) 
  - можна “розбудити” сигналом

  🔴 uninterruptible sleep (D) 
  - НЕ можна перервати сигналом
  - зазвичай очікує диск / I/O
  - часто проблема, якщо процес завис

- **4. stopped (зупинений)**

  👉 процес призупинений
  - не виконується
  - не планується

  Причини:
  - <kbd>Ctrl</kbd> + <kbd>Z</kbd>
  - сигнал `SIGSTOP`
  - дебагер

- **5. zombie (зомбі)**

  👉 процес вже завершився, але:
  - запис про нього ще є в системі
  - батьківський процес не забрав статус

  Що це означає
  - код виконання вже завершено
  - пам’ять звільнена
  - залишився лише запис у таблиці процесів

  Як виглядає
  ```
  Z
  ```

  Чому виникає

  Батьківський процес не викликав `wait()`

- **(додатково) orphan процес**

  👉 процес, у якого батько завершився

  автоматично переходить до PID 1 (systemd)

  🔥 Коротка таблиця

  | Стан | Значення              |
  | ---- | --------------------- |
  | R    | running / runnable    |
  | S    | interruptible sleep   |
  | D    | uninterruptible sleep |
  | T    | stopped               |
  | Z    | zombie                |

### 3.8 Контекст CPU процесу

Це одна з найважливіших частин внутрішньої роботи ОС.

🔹 Що таке контекст CPU

👉 це повний стан процесу, необхідний щоб:
- зупинити його
- зберегти
- потім відновити

🔹 Що входить у контекст

- **1. Регістри CPU**

  Це швидка пам’ять процесора:
  - загального призначення (RAX, RBX, RCX…)
  - використовуються для обчислень

- **2. Instruction Pointer (IP / RIP)**

  👉 вказує:
  - яку інструкцію виконувати далі

- **3. Stack Pointer (SP / RSP)**

  👉 вказує:
  - де вершина стеку

- **4. Flags (регістр стану)**

  👉 містить:
  - результат попередніх операцій
  - прапори (zero, carry, overflow…)

- **5. Program counter (по суті те ж, що IP)**
- **6. Контекст пам’яті**

  - mapping віртуальної пам’яті
  - таблиці сторінок

- **7. Стан FPU / SIMD**

  для обчислень з плаваючою точкою

- **8. Стан сигналів**
  - які сигнали очікуються
  - які маски сигналів
***

**Що таке context switch**

  👉 коли ядро:
  - зупиняє процес
  - зберігає його контекст
  - перемикається на інший процес
  - потім повертається назад
***

**Чому це дорого**

    Context switch:
    - не безкоштовний
    - займає CPU час
    - очищає кеші

    👉 тому багато процесів = overhead
***

🔥 Інтуїція

Уяви:
- CPU = один робітник
- процеси = задачі

Щоб переключитись:

  👉 треба запам’ятати все про поточну задачу  
  👉 і відновити іншу

### 3.9 Змінні оточення

Наприклад:
- PATH
- HOME
- USER
- SHELL

Вони є частиною середовища процесу.

### 3.10 Поточний робочий каталог

Процес знає, де він зараз “стоїть” у файловій системі.

Наприклад, якщо shell зараз у `/home/user/projects`, то запущені з нього команди зазвичай успадкують цей `working directory`.

### 3.11 Маска створення файлів (umask)

Процес має `umask`, яка впливає на права новостворених файлів і каталогів.

### 3.12 Обробники сигналів

Процес може:
- обробляти деякі сигнали власною функцією
- ігнорувати деякі сигнали
- використовувати стандартну поведінку

## 4. Формальне представлення процесу в ядрі

У Linux процес у ядрі описується структурою даних. На концептуальному рівні важливо зрозуміти:
> ядро для кожного процесу тримає запис, у якому є вся службова інформація про нього.

Часто в навчальних матеріалах це пояснюють як  `PCB` — Process Control Block.

У Linux це реалізовано через внутрішні kernel-структури, головна з яких історично пов’язана з task_struct.

Тобто ядро не “пам’ятає процес як ідею”, а має про нього конкретний запис у пам’яті, де є:
- PID
- стан
- пріоритет
- вказівники на батька/дітей
- таблиця файлів
- інформація про пам’ять
- сигналізація
- планування

Саме тому ядро може керувати тисячами процесів одразу.

## 5. PID — Process ID
**Що таке PID**

`PID` — це унікальний числовий ідентифікатор процесу.

Наприклад:
```bash
echo $$
```
У shell це покаже PID поточного shell-процесу.
***

**Навіщо потрібен PID**

PID використовується, щоб:
- знайти процес
- надіслати йому сигнал
- простежити його батьківство
- перевірити ресурси
- дебажити
- моніторити

Наприклад:
```bash
kill 1234
ps -p 1234
```
***
**Чи PID унікальний назавжди**

Ні.
Він унікальний у певний момент часу, але після завершення процесу PID може бути використаний знову.

Тобто PID не є вічним ідентифікатором “на всі часи”.
***

**PID 1**

У Linux особливий процес — `PID 1`.

Це перший користувацький процес, який запускається ядром після старту системи.
У сучасних дистрибутивах це зазвичай `systemd`.

PID 1 має особливу роль:
- запускає інші сервіси
- підхоплює осиротілі процеси
- є базовим коренем дерева процесів у user space

## 6. PPID — Parent Process ID
**Що таке PPID**

`PPID` — це PID батьківського процесу.

Майже кожен процес у Linux кимось породжений.

Наприклад:
- ти відкриваєш `terminal emulator`
- він запускає `shell`
- shell запускає `ls`

Тоді:
- ls має свій `PID`
- його `PPID` — це PID shell

Приклад
```bash
ps -o pid,ppid,cmd
```
Це покаже PID, PPID і команду.
***

**Навіщо потрібен PPID**

PPID допомагає:
- бачити ієрархію процесів
- розуміти, хто кого запустив
- дебажити
- знаходити процеси певного сервісу
- пояснювати orphan/zombie процеси

## 7. Батьківські та дочірні процеси
**Батьківський процес**

Процес, який створив інший процес, називається `батьківським`.

**Дочірній процес**

Створений ним новий процес — `дочірній`.

**Як це відбувається**

У класичній Unix/Linux-моделі створення процесу зазвичай відбувається так:
1. батьківський процес викликає fork()
2. ядро створює новий процес — копію батьківського
3. потім дочірній процес часто викликає exec() і запускає іншу програму

Приклад із shell

Коли ти пишеш:
```bash
ls -l
```
типовий сценарій такий:
1. bash читає команду
2. bash викликає `fork()`
3. створюється дочірній процес
4. у дочірньому процесі викликається `exec()` для запуску ls
5. ядро запускає `/bin/ls`
6. shell чекає завершення команди або продовжує роботу залежно від режиму

Чи всі процеси мають батька

Майже всі так.

Коренем користувацького дерева зазвичай є `PID 1`.

## 8. Дерево процесів

Оскільки майже кожен процес кимось створений, процеси утворюють ієрархію, або дерево процесів.

Як це виглядає концептуально
```
systemd(1)
 ├─ sshd
 │   └─ bash
 │       ├─ vim
 │       └─ python
 ├─ nginx
 └─ cron
```
Тут:
- systemd — предок
- sshd породив shell
- shell породив vim і python

Як подивитися дерево процесів
```bash
pstree
```
Або детальніше:
```bash
pstree -p
```
Тут будуть і назви процесів, і PID.

Навіщо розуміти дерево процесів

Це потрібно, щоб:
- розуміти, хто запустив процес
- бачити структуру сервісів
- дебажити
- відслідковувати ланцюжок запуску
- аналізувати orphan та daemon-процеси

## 9. init / systemd — перший процес користувацького простору
Що відбувається після запуску комп’ютера

Дуже спрощено ланцюжок такий:
1. вмикається комп’ютер
2. працює firmware: BIOS/UEFI
3. завантажується bootloader
4. bootloader завантажує ядро Linux
5. ядро ініціалізує пам’ять, драйвери, пристрої
6. ядро запускає перший процес user space

Ось цей перший процес історично називали init.

**Традиційний init**

У старих Unix/Linux системах першим процесом був `init`.

**Сучасний Linux**

У більшості сучасних дистрибутивів цим процесом є `systemd`, який має PID 1.

**Чому PID 1 особливий
**
PID 1:
- запускається ядром напряму
- є предком для більшості інших user-space процесів
- відповідає за запуск системних сервісів
- підхоплює orphan-процеси
- має особливу роль при завершенні системи

Як перевірити PID 1
```bash
ps -p 1 -f
```

## 10. Контекст процесу

Це дуже важливе поняття.

Що таке контекст процесу

`Контекст процесу` — це весь набір інформації, який потрібен ядру, щоб:
- зупинити процес
- зберегти його стан
- пізніше відновити його точно з того місця, де він був

**Що входить у контекст**

Спрощено:
- значення регістрів CPU
- instruction pointer
- stack pointer
- інформація про пам’ять
- стан планування
- відкриті файли
- права доступу
- маски сигналів
- інша службова інформація
***
**Навіщо це потрібно**

Бо CPU один і той самий (або кілька CPU-ядер), а процесів багато.

Тому Linux постійно:
- дає час одному процесу
- зупиняє його
- зберігає його контекст
- перемикається на інший
- потім повертається назад

Це називається context switch — перемикання контексту.
***
**Інтуїтивний приклад**

Уяви, що кілька людей пишуть твір, але ручка одна.

Щоб ніхто не загубився, потрібно кожного разу фіксувати:
- на якому слові зупинилися
- що вже написали
- що хотіли написати далі

Оце і є аналог контексту процесу.

## 11. Простір пам’яті процесу

Кожен процес має свій віртуальний адресний простір.

Це одна з фундаментальних ідей сучасних ОС.

**Що це означає**

Кожен процес “бачить” свою власну пам’ять як окремий простір адрес.

Тобто процес не працює напряму з “усією RAM комп’ютера”.
Він працює зі своїм віртуальним простором пам’яті, який ядро та MMU відображають на фізичну пам’ять.

**Чому це важливо**

Це дає:
 - ізоляцію процесів
 - безпеку
 - стабільність
 - зручність програмування

Один процес не повинен просто так читати чи ламати пам’ять іншого.
***
**Основні області пам’яті процесу**

Спрощено в адресному просторі процесу є:
- text segment — код програми
- data segment — глобальні/статичні дані
- bss — неініціалізовані глобальні/статичні змінні
- heap — динамічна пам’ять
- stack — стек викликів
- memory mapped regions — бібліотеки, mmap, shared objects
***

**Віртуальна пам’ять**

Коли процес звертається до адреси, він працює з віртуальною адресою, а не напряму з фізичною.

Ядро разом з апаратурою процесора перетворює віртуальні адреси на фізичні.

Завдяки цьому:
- кожен процес має відчуття власної ізольованої пам’яті
- однакові віртуальні адреси в різних процесах не конфліктують

ОС може гнучко розподіляти RAM
***
**Важливе уточнення**

“Процес має свою пам’ять” — це в повсякденному поясненні правильно.
Але технічно точніше так:
> процес має власний віртуальний адресний простір.

## 12. Чим процес відрізняється від job

Це дуже важливе розрізнення.

**Процес**

`Процес` — це об’єкт ядра.

Ядро:
- створює процес
- планує його
- зберігає його стан
- дає PID
***

**Job**

`Job` — це поняття оболонки, наприклад bash.

Shell може групувати один або кілька процесів у “завдання”, щоб зручно керувати ними:
- запускати у фоні
- повертати у foreground
- зупиняти
- продовжувати
***

**Чому це не одне й те саме**

Наприклад:
```bash
sleep 100 &
```
- для ядра це просто процес
- для shell це job номер [1]

Shell може показати:
```
jobs
```
Але ядро не оперує “job 1”, воно оперує PID, process groups, signals.

**Важливий наслідок**

Не можна змішувати:
- процеси — це рівень ядра
- jobs — це рівень shell

Окремий розділ про jobs ми ще зробимо пізніше.

## 13. Чим процес відрізняється від потоку

Ще одна фундаментальна різниця.

**Процес**

Процес має:
- окремий адресний простір
- свої ресурси
- свої відкриті файли
- свої атрибути безпеки
- власний контекст виконання

**Потік**

`Потік` — це окрема лінія виконання в межах процесу.

Кілька потоків одного процесу зазвичай:
- ділять одну пам’ять
- ділять відкриті файли
- ділять ресурси процесу

Але кожен потік має свої:
- регістри
- стек
- instruction pointer
- стан виконання

**Просте порівняння**
- процес = окрема квартира
- потоки = кілька людей усередині цієї квартири

Усередині вони користуються спільними речами, але кожен виконує свою роботу.

**Чому це важливо**

Бо новачки часто плутають:
- threads — потоки виконання
- stdin/stdout streams — потоки вводу/виводу

Це зовсім різні речі.

## 14. Як подивитися процеси в Linux

На базовому рівні корисно вже тут знати кілька команд.

- Подивитися поточні процеси
  ```bash
  ps
  ```
  Це покаже процеси поточного shell-сеансу.

- Подивитися більше інформації
  ```bash
  ps aux
  ```
  або в сучаснішому стилі:
  ```bash
  ps -ef
  ```
- Подивитися дерево процесів
  ```bash
  pstree
  ```
  або:
  ```bash
  pstree -p
  ```
- Подивитися інтерактивно
  ```bash
  top
  ```
  або, якщо встановлено:
  ```bash
  htop
  ```
- Подивитися свій PID у shell
  ```bash
  echo $$
  ```
  Подивитися PID 1
  ```bash
  ps -p 1 -f
  ```

## 15. Що ще важливо зрозуміти вже зараз
**Процес — це не просто “вікно програми”**

Наприклад, у фоновому режимі можуть працювати процеси, які ти не бачиш як вікно:
- sshd
- cron
- systemd
- dbus-daemon
- nginx

**Один сервіс може мати кілька процесів**

Наприклад:
- master process
- worker processes
- helper processes

Тому “одна програма” не завжди означає “один процес”.

**Один користувач може мати багато процесів**

Наприклад, твій shell, редактор, браузер, IDE, Python-скрипти — усе це різні процеси.

**Процеси ізольовані**

У нормі один процес не повинен напряму змінювати пам’ять іншого.
Це критично для стабільності ОС.

## 16. Найважливіші терміни цього розділу
| Поняття             | Визначення                                                           |
| ------------------- | -------------------------------------------------------------------- |
| Процес              | Запущена програма, якою керує ядро Linux.                            |
| Програма            | Файл на диску з інструкціями, ще не обов’язково запущений.           |
| PID                 | Унікальний ідентифікатор процесу.                                    |
| PPID                | Ідентифікатор батьківського процесу.                                 |
| Батьківський процес | Процес, який створив інший.                                          |
| Дочірній процес     | Процес, створений іншим процесом.                                    |
| Дерево процесів     | Ієрархія процесів за зв’язком parent-child.                          |
| PID 1               | Перший процес user space, зазвичай systemd.                          |
| Контекст процесу    | Уся службова інформація, потрібна для зупинки й відновлення процесу. |
| Адресний простір    | Віртуальна пам’ять, у якій працює процес.                            |
| Job                 | Поняття shell для керування командами у foreground/background.       |
| Потік               | Окрема лінія виконання всередині процесу.                            |

## 17. Короткий підсумок розділу

Головна ідея така:
- програма — це пасивний файл на диску
- процес — це активне виконання програми
- процес існує як об’єкт ядра
- у нього є PID, пам’ять, відкриті файли, права, стан і контекст
- майже кожен процес має батька
- усі процеси утворюють дерево
- коренем user-space дерева зазвичай є PID 1 (systemd)
- процес не те саме, що job
- процес не те саме, що thread

## 18. Міні-перевірка розуміння

Ось кілька контрольних тез. Якщо вони зрозумілі, значить база вже закладена правильно:

1. Одна програма може бути запущена багато разів, і це будуть різні процеси.
2. Процес має PID, а програма як файл — ні.
3. Процесом керує ядро, а job-ами — shell.
4. Кожен процес має свій віртуальний адресний простір.
5. Процеси зазвичай утворюють дерево parent-child.
6. PID 1 — особливий процес у системі.
7. Потік — це не те саме, що процес.
