## 1. Користувацькі процеси (User Processes)
**🔹 Що це таке**

👉 Це процеси, які працюють у user space (користувацькому просторі)
- запускаються користувачем або сервісами
- не мають прямого доступу до ядра
- для доступу до ресурсів використовують системні виклики (syscalls)

**🔹 Приклади**
- bash
- python
- node
- nginx
- vim
- браузери
- твої програми

**🔹 Властивості**
| Характеристика          | Опис                                  |
| ----------------------- | ------------------------------------- |
| Працюють у user space   | не мають прямого доступу до kernel    |
| Ізольовані              | не можуть напряму ламати систему      |
| Використовують syscalls | щоб читати файли, працювати з мережею |
| Мають UID/GID           | працюють від імені користувача        |

**🔹 Важливий момент**

👉 user process не може напряму:
- працювати з апаратурою
- керувати пам’яттю ядра
- змінювати kernel

Усе через ядро.

## 2. Процеси ядра (Kernel Processes / Kernel Threads)
**🔹 Що це таке**

👉 Це процеси, які працюють всередині ядра Linux
- їх створює ядро
- вони виконують системні задачі
- не мають user space

**🔹 Як їх побачити**
```bash
ps -ef | grep '\['
```
або:
```bash
ps aux
```
Ти побачиш щось типу:
```
[kthreadd]
[kworker/0:1]
[rcu_sched]
```

**🔹 Ознака**

👉 їх назви в квадратних дужках:
```
[kworker]
```

**🔹 Властивості**
| Характеристика                       | Опис                     |
| ------------------------------------ | ------------------------ |
| Працюють у kernel space              | повний доступ до системи |
| Не мають user memory                 | не як звичайні процеси   |
| Не запускаються користувачем         | створюються ядром        |
| Часто не мають звичайного executable | немає /bin/...           |

**🔹 Приклади**
- `[kthreadd]` — менеджер kernel threads
- `[kworker]` — фонова робота ядра
- `[migration]` — балансування CPU
- `[rcu_*]` — механізми синхронізації

**🔹 Важливо**

👉 це не зовсім “процеси” в класичному сенсі  
👉 це kernel threads

**🔥 Чому вони існують**

Бо ядро теж має виконувати задачі:
- обробка I/O
- планування
- робота драйверів
- обробка переривань

## 3. Демони (Daemon Processes)
**🔹 Що таке демон**

👉 це процес, який працює у фоні і виконує системну або сервісну задачу

**🔹 Ознаки**
- працює без взаємодії з користувачем
- часто запускається при старті системи
- має довгий життєвий цикл
- зазвичай не має terminal

**🔹 Приклади**
- `sshd` — SSH сервер
- `nginx` — вебсервер
- `cron` — планувальник задач
- `systemd` — менеджер сервісів
- `dbus-daemon`

**🔹 Чому "daemon"**

Історично:

👉 daemon = “службовий процес, який працює у фоні”

**🔹 Як вони працюють**

Часто:
- запускаються systemd
- від’єднуються від terminal
- працюють у фоні
- слухають події (мережа, таймери, запити)
***

**🔹 daemon vs звичайний процес**
| Звичайний процес    | Демон             |
| ------------------- | ----------------- |
| короткоживучий      | довгоживучий      |
| має terminal        | зазвичай ні       |
| запускається вручну | часто автоматично |

## 4. Інтерактивні vs неінтерактивні процеси
**🔹 Інтерактивні**

👉 взаємодіють з користувачем

Приклади:
- `bash`
- `vim`
- `nano`
- `top`

**🔹 Неінтерактивні**

👉 працюють без участі користувача
- скрипти
- демони
- cron jobs

**🔥 Різниця**
| Тип            | Особливість   |
| -------------- | ------------- |
| Інтерактивні   | чекають вводу |
| Неінтерактивні | працюють самі |

## 5. Foreground vs Background

👉 це вже ближче до shell, але важливо тут

**🔹 Foreground процес**

👉 процес, який:
- займає terminal
- отримує ввід з клавіатури

Приклад:
```bash
nano file.txt
```

**🔹 Background процес**

👉 працює у фоні
```bash
sleep 100 &
```

**🔹 Важливо**

- це не тип процесу ядра
- це режим роботи через shell

## 6. CPU-bound vs I/O-bound процеси

Це вже про поведінку процесу.

**🔹 CPU-bound**

👉 процес, який активно використовує CPU

Приклади:
- обчислення
- шифрування
- рендеринг
- алгоритми

**🔹 I/O-bound**

👉 процес, який більшість часу чекає
- диск
- мережа
- база даних

**🔥 Різниця**

| Тип       | Чим зайнятий |
| --------- | ------------ |
| CPU-bound | обчислення   |
| I/O-bound | очікування   |

**🔹 Чому це важливо**

Scheduler поводиться по-різному:
- CPU-bound → довше виконується
- I/O-bound → часто "прокидається"

## 7. Як розпізнати тип процесу
**🔹 Kernel vs user**
```bash
ps aux
```
👉 kernel → [name]

**🔹 CPU usage**
```bash
top
```
👉 якщо 100% CPU → CPU-bound

**🔹 Інтерактивність**

- має terminal → інтерактивний
- не має → daemon або background

**🔹 Перевірити TTY**
```bash
ps -o pid,tty,cmd
```

## 8. Важливі нюанси
❗ Kernel thread ≠ user process

- kernel thread → частина ядра
- user process → окремий процес

❗ daemon — це user process

👉 просто особливий тип (фоновий)

❗ background ≠ daemon
  ```bash
  sleep 100 &
  ```
👉 це background, але НЕ daemon

❗ один процес може змінювати поведінку

- бути foreground → потім background
- бути інтерактивним → потім ні
