## Що це таке

`OOM Killer` (Out Of Memory Killer — механізм завершення процесів при нестачі пам’яті)

👉 Це механізм ядра Linux, який:

коли закінчується RAM → примусово вбиває процеси

## 2 Коли він запускається

OOM Killer активується коли:
```
немає доступної пам’яті
+
немає можливості виділити нову
```

🔍 Причини
- RAM заповнена
- swap теж заповнений або відключений
- memory leak (витік пам’яті)
- дуже великий процес
- overcommit memory
  
## 3 Як Linux вирішує, кого вбити

Linux використовує:

> OOM score (оцінка для вбивства)

**📊 OOM score**

Кожен процес має:
- oom_score (оцінка для завершення)
- чим більше значення → тим більша ймовірність бути вбитим

**Подивитись**
```bash
cat /proc/<PID>/oom_score
```

**📌 Вплив**

OOM score залежить від:
- скільки RAM використовує процес
- чи це system process (системний процес)
- чи має root privileges (права root)
- чи має спеціальні прапорці

## 4 oom_score_adj

`oom_score_adj` (коригування OOM score — пріоритет захисту/знищення)

📌 Діапазон
- `-1000` → ніколи не вбивати
- `0` → стандарт
- `+1000` → вбити першим
  
📌 Подивитись
```bash
cat /proc/<PID>/oom_score_adj
```

📌 Змінити
```bash
echo -1000 > /proc/<PID>/oom_score_adj
```

📌 Приклад
-1000 → критичні сервіси (systemd)
+500 → фонові процеси

## 5 Як виглядає OOM Killer в логах
```bash
dmesg | grep -i oom
```

📌 Приклад  
```
Out of memory: Kill process 1234 (node) score 987
Killed process 1234 (node)
```
🔍 Що означає
> Kill process 1234 → кого обрали  
> score 987 → чому саме його

## 6 Що відбувається під капотом
1. процес просить пам’ять
2. ядро не може виділити
3. запускається OOM Killer
4. обирається “жертва”
5. процес отримує SIGKILL
6. пам’ять звільняється
   
## 7 Чому це небезпечно

OOM Killer може вбити:
- базу даних
- backend
- critical service

👉 і ти просто побачиш:

> процес зник

## 10 Overcommit memory

Linux дозволяє:

> overcommit (перевиділення пам’яті)

📌 Параметр
```bash
cat /proc/sys/vm/overcommit_memory
```

📊 Значення
| Значення | Опис                         |
| -------- | ---------------------------- |
| 0        | heuristic (за замовчуванням) |
| 1        | завжди дозволяти             |
| 2        | суворо обмежувати            |

## 11 Найважливіше

Якщо процес “раптово зник”

перевір:
```bash
dmesg | grep -i kill
```
Якщо бачиш
> Killed process ...

👉 це OOM Killer

## 13 Що відбувається після OOM

Коли спрацьовує  
> OOM Killer (Out Of Memory Killer — механізм завершення процесів при нестачі пам’яті)

👉 відбувається:

1. Ядро обирає процес
2. Надсилає SIGKILL (примусове завершення)
3. Процес миттєво завершується
4. Пам’ять звільняється
  
❗ Важливо  
- процес не можна врятувати
- він не має шансу коректно завершитись
- дані можуть бути втрачені

## 14 Що робити, якщо вбили важливий процес
✅ Крок 1: підтвердити причину
```bash
dmesg | grep -i oom
```
або:
```bash
dmesg | grep -i "killed process"
```
✅ Крок 2: зрозуміти, хто винен

Подивись:

> Killed process 1234 (node) score 987

👉 це означає:
- node використовував багато пам’яті
- або мав високий oom_score

✅ Крок 3: перевірити систему`
```bash
free -h
vmstat 1
```

✅ Крок 4: запустити сервіс знову

якщо це:
- systemd service → він може піднятись сам
- docker container → залежить від restart policy

## 15 Чи працює OOM автоматично?

👉 так, повністю автоматично

Це частина ядра Linux.

📌 Важливо
> OOM Killer = last resort (останній механізм захисту)

Він спрацьовує тільки коли:

- RAM закінчилась
- swap не допомагає
- ядро не може виділити пам’ять

## 16 Чи можна налаштувати OOM?

👉 так, і це дуже важливо в продакшені

**oom_score_adj**

📌 Що це

`oom_score_adj` (коригування ймовірності бути вбитим)

📊 Діапазон
```
-1000 → NEVER kill (ніколи не вбивати)
0 → default
+1000 → kill first (вбити першим)
```

📌 Приклад
```bash
cat /proc/<PID>/oom_score_adj
```

📌 Встановити
```bash
echo -1000 > /proc/<PID>/oom_score_adj
```

🔥 Практика
| Тип процесу          | Значення |
| -------------------- | -------- |
| PostgreSQL           | -900     |
| Backend API          | -500     |
| Cache                | 0        |
| Batch jobs           | +300     |
| Non-critical scripts | +500     |

## 17. Як захистити важливі процеси
### 17.1 Захист через oom_score_adj

👉 найпростіший і обов’язковий спосіб

### 17.2 systemd налаштування
📌 У service-файлі
```
[Service]
OOMScoreAdjust=-1000
```
👉 це означає:

> “ніколи не вбивати цей сервіс”

### 17.3 Використання swap
📌 перевірити
```bash
swapon --show
```
❗ якщо swap = 0

👉 OOM буде спрацьовувати дуже часто

### 17.4 Обмеження пам’яті (cgroups / Docker)
Docker приклад
```bash
docker run -m 512m myapp
```
👉 це запобігає ситуації, коли один процес “з’їдає все”

### 17.5 Моніторинг

👉 потрібно ловити проблему ДО OOM
```bash
free -h
vmstat
top
```
або:
- Prometheus
- Grafana
- alerts

## 18 Що НЕ можна робити
❌ Ставити всім oom_score_adj = -1000

👉 тоді:

OOM не зможе нікого вбити → система зависне

❌ Ігнорувати swap

👉 без swap:
- система дуже нестабільна
- OOM частіший
  
## 19 Як мислити правильно

🔥 Головна ідея
> OOM = симптом, а не причина

**Причини:**
- memory leak
- неправильні ліміти
- погана архітектура
- недостатньо RAM
  
## 20 Типовий сценарій
```
Сайт впав
→ процес зник
→ dmesg
→ Killed process node
→ значить OOM
→ дивимось RAM
→ бачимо 100% usage
→ причина: memory leak
```

## 21 Що відбувається в реальному продакшені
Без налаштування:
> OOM вбиває випадковий процес 😈

З правильним налаштуванням:
> OOM вбиває найменш важливий процес 👍

## 22 Де подивитись oom_score

### 22.1 Для конкретного процесу
```bash
cat /proc/<PID>/oom_score
```
📌 Приклад
```bash
cat /proc/1234/oom_score
```
Вивід:

> 987

👉 Чим більше число → тим більша ймовірність, що процес вб’є
OOM Killer (Out Of Memory Killer — механізм завершення процесів при нестачі пам’яті)

### 22.2 Як подивитись для ВСІХ процесів
✅ Варіант 1 (найпростіший)
  ```bash
  for pid in $(ls /proc | grep -E '^[0-9]+$'); do 
    echo -n "$pid "
    cat /proc/$pid/oom_score 2>/dev/null
  done | sort -k2 -n
  ```

📌 Що це робить
- проходиться по всіх PID
- читає oom_score
- сортує по значенню
  
📊 Результат
```
123   10
456   50
789   900
```
👉 внизу будуть “жертви” (найвищий score)

✅ Варіант 2 (з назвою процесу)
```bash
for pid in $(ls /proc | grep -E '^[0-9]+$'); do 
  score=$(cat /proc/$pid/oom_score 2>/dev/null)
  cmd=$(tr -d '\0' < /proc/$pid/cmdline 2>/dev/null)
  echo "$pid $score $cmd"
done | sort -k2 -n
```
📌 Вивід
```
123 10 /usr/sbin/sshd
456 200 /usr/bin/python app.py
789 950 node server.js
```
👉 одразу видно:
- PID
- score
- команда
  
✅ Варіант 3 (top-like вигляд)
```bash
ps -eo pid,comm --no-headers | while read pid cmd; do 
  score=$(cat /proc/$pid/oom_score 2>/dev/null)
  echo "$pid $score $cmd"
done | sort -k2 -nr | head
```

📊 Вивід
```
789 950 node
456 800 python
321 700 chrome
```
👉 топ кандидатів на “вбивство”

### 22.3 Де подивитись oom_score_adj
📌 Команда
```bash
cat /proc/<PID>/oom_score_adj
```
📊 Приклад
```
-1000 → не вбивати
0 → стандарт
+500 → бажано вбити
```

### 22.4 Комбінований варіант (дуже корисно)
```bash
for pid in $(ls /proc | grep -E '^[0-9]+$'); do 
  score=$(cat /proc/$pid/oom_score 2>/dev/null)
  adj=$(cat /proc/$pid/oom_score_adj 2>/dev/null)
  cmd=$(tr -d '\0' < /proc/$pid/cmdline 2>/dev/null)
  echo "$pid score=$score adj=$adj $cmd"
done | sort -k2 -nr | head
```
📊 Вивід
```
789 score=950 adj=0 node server.js
456 score=800 adj=0 python app.py
123 score=50 adj=-1000 postgres
```
👉 одразу видно:
- хто під ризиком
- хто захищений

**⚠️ Важливі нюанси**  
❗ 1. oom_score змінюється

👉 це не статичне значення

залежить від:
  - використання пам’яті
  - стану системи
  - інших процесів

❗ 2. root-процеси мають менший шанс

Linux намагається не вбивати критичні процеси

❗ 3. cmdline може бути пустим

особливо для:
- kernel threads
- короткоживучих процесів

### 22.5 Як це використовувати в реальності
🔥 Якщо боїшся OOM
```bash
# знайти кандидатів
... | sort -k2 -nr | head
```

🔥 Якщо щось вже впало
```bash
dmesg | grep -i "killed process"
```
і порівняти з тим, хто мав великий score
