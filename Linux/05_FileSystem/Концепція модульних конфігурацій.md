# Концепція модульних конфігурацій (Директорії .d)

## 1. Що це таке?
Директорії, що закінчуються на .d (від англ. directory), призначені для зберігання фрагментів конфігурації. Замість того, щоб редагувати один величезний файл (наприклад, /etc/sudoers), система зчитує головний файл, а потім автоматично додає до нього всі маленькі файли з відповідної .d папки.

## 2. Як це працює технічно?
У головному файлі зазвичай є спеціальна директива include.
Наприклад, у `/etc/sudoers` ви знайдете рядок:
```
@includedir /etc/sudoers.d
```
Це команда системі: «Після того, як прочитаєш цей файл, піди в папку /etc/sudoers.d і виконай усе, що там лежить».

## 3. Чому це важливо (Переваги):
- Безпека оновлень (Package Management): Коли ви оновлюєте систему, пакетний менеджер (APT/YUM) замінює головний файл. Ваші особисті файли в `.d` папках залишаються недоторканими.
- Уникнення помилок: Якщо ви припуститеся помилки в окремому файлі (наприклад, `/etc/sudoers.d/my_rules`), ви зламаєте лише ці конкретні права, а не весь механізм sudo.
- Автоматизація: Програми (Docker, Nginx) можуть самі додавати свої налаштування під час встановлення, не "колупаючись" у ваших ручних правках.
- Порядок: Легко знайти, де саме ви прописали доступи для групи universe, не гортаючи 500 рядків тексту.

## 4. Правила іменування файлів у .d:
Щоб система "побачила" файл у такій папці, він зазвичай має відповідати правилам:
- Без розширень: У багатьох випадках (як у sudoers.d) файли не повинні мати крапок у назві (наприклад, rules — добре, rules.conf — може бути проігноровано).
- Алфавітний порядок: Система читає файли за алфавітом. Файл 01_start виконається раніше, ніж 99_final. Це важливо для правил, які можуть перекривати одне одного.
- Тимчасові файли: Файли, що закінчуються на ~ або .dpkg-old, автоматично ігноруються.

## 5. Приклади найважливіших .d папок:
| Папка                    | Для чого використовується                        |
| ------------------------ | ------------------------------------------------ |
| /etc/sudoers.d/          | Права користувачів та груп (sudo)                |
| /etc/apt/sources.list.d/ | Адреси нових репозиторіїв (де качати софт)       |
| /etc/sysctl.d/           | Тонкі налаштування ядра (мережа, пам'ять)        |
| /etc/cron.d/             | Планувальник завдань (запуск за розкладом)       |
| /etc/profile.d/          | Налаштування термінала (аліаси, змінні оточення) |

## 6. сценарії використання директорій з .d
у Linux існує два сценарії використання директорій з .d:
- Доповнення (як у випадку з sudoers + sudoers.d): коли є "батьківський" файл і папка для розширень.
- Чиста модульність (наприклад depmod.d/ чи binfmt.d/): коли є тільки папка, а єдиного головного файлу в /etc не існує.

**Чому так відбувається?**
У другому випадку головний (дефолтний) конфіг зазвичай "схований" глибоко в системних папках (наприклад, у /usr/lib/), щоб користувач його випадково не зіпсував.  
Ось як працює ця логіка (її варто додати до вашого розділу про пріоритети):  
- Системний рівень (/usr/lib/...): там лежать заводські налаштування від розробників.
- Адміністративний рівень (/etc/... або /etc/...d/): тут ви створюєте свої файли.
  
Коли програма (наприклад, depmod) запускається, вона перевіряє: "Чи створив адмін щось у /etc/depmod.d/? Якщо так — беру ці налаштування. Якщо там порожньо — беру стандартні з системи".

**Закономірність:**
- Якщо файл + папка .d: Папка потрібна для зручного додавання правил, щоб не псувати основний файл.
- Якщо тільки папка .d: Це означає, що система повністю перейшла на модульний формат. Ви просто кидаєте туди файл, і він стає активним.

> [!TIP]
> Цікавий факт: У сучасних дистрибутивах Linux (як-от Fedora або Arch) розробники намагаються взагалі прибрати файли з `/etc`, залишаючи лише папки `.d`, щоб система була максимально чистою після встановлення.


## 7. Таблиця пріоритетів конфігурацій
Вона пояснює золоте правило Linux: «Хто останній за алфавітом або ближче до користувача — той і головний».

**Таблиця пріоритетів конфігурацій (Hierarchy of overrides)**
Ця логіка працює для більшості системних служб (systemd, udev, sysctl, sudoers тощо).
| Рівень пріоритету | Розташування (Шлях)                            | Хто зазвичай створює                                                                     | Опис і логіка                                                                                                          |
| ----------------- | ---------------------------------------------- | ---------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| 1. Найвищий       | `/etc/назва.d/99-custom.conf`                  | Адміністратор (Ви)                                                                       | Файли в `/etc` з великими цифрами в назві (наприклад, 99-) перекривають усе інше.                                      |
| 2. Високий        | `/etc/назва.d/01-base.conf`	Адміністратор (Ви) | Ваші файли з меншими номерами. Вони "головніші" за системні, але "слабші" за ваші ж 99-. |
| 3. Середній       | `/etc/назва.conf`                              | Система / Інсталятор                                                                     | Основний (монолітний) файл. Зазвичай читається першим, але може бути доповнений через `.d`.                            |
| 4. Низький        | `/run/назва.d/`                                | Процеси (Runtime)                                                                        | Тимчасові налаштування, що існують лише до перезавантаження (створюються скриптами).                                   |
| 5. Найнижчий      | `/usr/lib/назва.d/`                            | Розробники (Vendor)                                                                      | "Заводські" налаштування. Якщо ви створите файл з таким же іменем у `/etc`, системний файл буде повністю ігноруватися. |

**3 головних правила "виживання" в конфігах:**
- Алфавітний порядок (Lexicographical order):  
Всередині папки .d файли читаються по черзі: `01-net.conf` -> `02-net.conf` -> `10-net.conf`. Якщо в першому файлі написано "IP=10.1.1.1", а в останньому "IP=192.168.1.1", система обере останній варіант.

- Маскування (Masking):  
Якщо ви хочете повністю "вимкнути" системне налаштування з `/usr/lib/назва.d/test.conf`, просто створіть порожній файл (або посилання на `/dev/null`) з таким же ім'ям у `/etc/назва.d/test.conf`. Система побачить файл у `/etc` і навіть не загляне в `/usr/lib`.

- Права доступу:  
Більшість `.d` папок вимагають прав root (права `0644` для читання або `0440` для sudoers). Якщо ви створите файл, який система не зможе прочитати, вона його просто проігнорує (іноді навіть без попередження).
