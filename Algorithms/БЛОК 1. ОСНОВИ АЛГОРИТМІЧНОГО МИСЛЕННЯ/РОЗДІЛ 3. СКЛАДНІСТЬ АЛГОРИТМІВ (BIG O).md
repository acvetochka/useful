## 3.1. Проблема, яку вирішує Big O

Уявімо дві програми, що виконують одну задачу.

| Алгоритм |	Час |
|---------| ------| 
| A	| 0.01 сек |
| B	| 10 хв |

Обидва правильні, але один практично непридатний.

Потрібен спосіб відповісти:

> наскільки добре масштабується алгоритм при зростанні даних?

Саме це і робить Big O.

## 3.2. Що таке складність алгоритму

Складність алгоритму — це залежність кількості операцій від розміру вхідних даних.

Позначення:

- `n` — кількість даних

Наприклад:
```
n = 10 елементів

n = 1 000

n = 1 000 000
```

## 3.3. Чому не вимірюють секунди

Час виконання залежить від:
- процесора
- мови програмування
- оптимізацій компілятора
- навантаження системи

Тому замість секунд рахують:

> 👉 кількість операцій.

## 3.4. Ідея Big O

Big O показує:

> як швидко росте кількість операцій при збільшенні n.

## 3.5. Найпростіший приклад
```python
for i in range(n):
    print(i)
```
Цикл виконується `n` разів.

Кількість операцій:
  ```
  ≈ n
  ```
Складність:
  ```
  O(n)
  ```
  Читається: "О від n".

## 3.6. Константна складність — O(1)
```python
x = arr[5]
```
Незалежно від розміру масиву:
- 10 елементів
- 1 мільйон
  
операція виконується однаково швидко.
```
O(1)
```
👉 найкращий можливий випадок.

## 3.7. Лінійна складність — O(n)
```python
for item in arr:
    if item == target:
        return True
```
У найгіршому випадку перевіряємо всі елементи.
```
O(n)
```
Подвоїли дані → подвоївся час.

## 3.8. Квадратична складність — O(n²)
```python
for i in arr:
    for j in arr:
        print(i, j)
```
Якщо:

- n = 5 → 25 операцій
- n = 100 → 10 000 операцій

Складність:
```
O(n²)
```
👉 дуже швидко стає повільним.

## 3.9. Логарифмічна складність — O(log n)

Найважливіша ідея.

Приклад — бінарний пошук.

Замість перевірки кожного елемента:
- беремо середину;
- відкидаємо половину даних;
- повторюємо.

Кількість перевірок:
```
n → n/2 → n/4 → n/8 ...
```
Складність:
```
O(log n)
```
Навіть при мільйоні елементів потрібно ≈ 20 кроків.

👉 надзвичайно швидко.

## 3.10. Логлінійна складність — O(n log n)

Типово для ефективних сортувань:
- Merge Sort
- Quick Sort

Поєднання:
- проходу по всіх елементах
- логарифмічного поділу.

## 3.11. Експоненційна складність — O(2ⁿ)
```
fibonacci(n)
```
Кожен виклик породжує два нових.
```
n = 10 → 1024 операцій
n = 20 → 1 000 000+
```
👉 практично непридатно.

## 3.12. Порівняння складностей
| Big O |	Назва	 | Якість |
| ----- |  ----- | ----- |
|O(1)	| константна	| ⭐⭐⭐⭐⭐ |
| O(log n)| 	логарифмічна |	⭐⭐⭐⭐⭐|
| O(n)	| лінійна	| ⭐⭐⭐⭐| 
| O(n log n)| 	добра	| ⭐⭐⭐| 
| O(n²)	| повільна	| ⭐⭐| 
| O(2ⁿ)	| дуже погано	| ⭐| 

## 3.13. Як рахувати Big O (правила)
- Правило 1. Ігноруємо константи
  ```
  O(2n) → O(n)
  O(100n) → O(n)
  ```
  Причина: при великих n різниця неважлива.

- Правило 2. Беремо найшвидше зростаючу частину
  ```
  O(n² + n) → O(n²)
  ```
- Правило 3. Вкладені цикли множаться
  ```
  for i in range(n):
    for j in range(n):
  ```
  ```
  O(n * n) = O(n²)
  ```
- Правило 4. Послідовні блоки додаються
  ```
  loop n
  loop n
  ```
  ```
  O(n + n) = O(n)
  ```

## 3.14. Best / Average / Worst Case

Алгоритм може працювати по-різному.

- **Best case** - Найкращий випадок.
- **Average case** - Середній.
- **Worst case** - Найгірший (зазвичай його і аналізують).

Приклад пошуку:
 - елемент перший → O(1)
 - елемент останній → O(n)

## 3.15. Чому Big O — це головна мова програмістів

Senior-розробники мислять так:
- структура даних → складність
- алгоритм → складність
- операція → складність

Саме тому:
- HashMap популярні
- Binary Search важливий
- правильні структури даних критичні

## 3.16. Найважливіший інсайт

Алгоритми — це боротьба з ростом даних.

Маленькі дані приховують проблеми.

Великі дані їх показують.

## Підсумок розділу

Big O описує:

- як алгоритм масштабується;
- як росте кількість операцій;
- наскільки рішення придатне для великих даних.
