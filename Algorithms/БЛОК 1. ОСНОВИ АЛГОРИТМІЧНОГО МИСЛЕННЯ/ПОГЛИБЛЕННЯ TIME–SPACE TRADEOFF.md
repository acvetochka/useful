📘 ПОГЛИБЛЕННЯ: TIME–SPACE TRADEOFF

# 1. Формальне визначення

**Time–Space Tradeoff** — це ситуація, коли зменшення часу виконання алгоритму призводить до збільшення використання пам’яті (або навпаки).

Математично:

>  Зменшуємо T(n) → зростає S(n)
> 
>  Зменшуємо S(n) → зростає T(n)

де

- T(n) — часова складність
- S(n) — просторова складність

## 2. Чому цей компроміс взагалі існує?

Бо комп’ютер має два ресурси:
- CPU (час обчислення)
- RAM (памʼять)

Щоб пришвидшити обчислення, ми часто:
- зберігаємо проміжні результати
- створюємо індекси
- будуємо додаткові структури
- кешуємо значення

Тобто ми замінюємо обчислення збереженням.

## 3. Ключовий принцип

Час можна зменшити двома способами:
1. Робити менше обчислень.
2. Не повторювати обчислення.

Другий спосіб майже завжди означає — зберігати результати.

А зберігання = памʼять.

## 4. Глибокий приклад №1 — перевірка повторів
Задача:

> Чи є в масиві дублікати?

Варіант 1 — подвійний цикл
```python
for i in range(n):
    for j in range(i+1, n):
        if arr[i] == arr[j]:
            return True
```
Аналіз:
- Час: O(n²)
- Памʼять: O(1)

Ми нічого не зберігаємо.

Варіант 2 — використати множину
```python
seen = set()
for x in arr:
    if x in seen:
        return True
    seen.add(x)
```
Аналіз:
- Час: O(n)
- Памʼять: O(n)

Ми зменшили час у тисячі разів,
але створили додаткову структуру.

## 5. Глибокий приклад №2 — Фібоначчі
**Наївна рекурсія**
```python
def fib(n):
    if n <= 1:
        return n
    return fib(n-1) + fib(n-2)
```
Час:
```
O(2ⁿ)
```
Памʼять:
```
O(n) (через стек)
```

**Мемоізація (запам’ятовування результатів)**
```python
memo = {}

def fib(n):
    if n in memo:
        return memo[n]
    if n <= 1:
        return n
    memo[n] = fib(n-1) + fib(n-2)
    return memo[n]
```
Час:
```
O(n)
```
Памʼять:
```
O(n)
```
Ми:
- зменшили час з експоненційного до лінійного
- використали памʼять для зберігання результатів

Це класичний time–space tradeoff.

## 6. Індексування — фундамент баз даних

Уявімо таблицю з мільйоном записів.

- Без індексу

  Пошук:

  `O(n)`

- З індексом (B-Tree / Hash index)

  Пошук:

  `O(log n) або O(1)`

Але індекс займає памʼять.

Тобто база даних жертвує памʼяттю заради швидкого доступу.

## 7. Кешування (Caching)

Приклад: веб-сервер.

Якщо кожен запит обчислювати заново:
- багато CPU часу
- повільна відповідь

Рішення:
- зберігати результат в кеші
- віддавати з памʼяті

Швидкість ↑

Памʼять ↑

## 8. Коли вигідніше економити памʼять?

Є системи, де памʼять критична:
- embedded devices
- мікроконтролери
- IoT
- старі смартфони
- високонавантажені сервери

Там часто обирають:
- повільніший алгоритм
- але без додаткових структур

## 9. Коли вигідніше економити час?

- API з жорстким SLA
- фінансові транзакції
- high-frequency trading
- обробка великих даних
- real-time системи

Там памʼять дешевша за затримку.

## 10. Неочевидний момент

Іноді додаткова памʼять також:
- покращує кеш CPU
- зменшує випадкові звернення до RAM
- робить доступ більш локальним

Тобто памʼять може навіть непрямо пришвидшувати виконання.

## 11. Глибший принцип: Перетворення часу в памʼять

Можна сказати так:

> **Памʼять** — це спосіб зберегти вже виконану роботу.

Кожен раз, коли ти:
- кешуєш
- будуєш таблицю
- створюєш мапу
- запамʼятовуєш проміжний результат
- ти купуєш швидкість за рахунок RAM.

## 12. А чи існує протилежний tradeoff?

**Так.**

Іноді можна:
- зменшити памʼять
- але перераховувати значення кожен раз

Наприклад:

Замість зберігання всіх префіксних сум —
обчислювати їх щоразу.

## 13. Реальна стратегія прийняття рішення

При виборі алгоритму задають:
1. Який розмір n?
2. Чи повторюється операція багато разів?
3. Чи є обмеження памʼяті?
4. Чи критичний час відповіді?
5. Чи масштабуватиметься система?

## 14. Найглибший інсайт

>**Time–space tradeoff** — це не технічна деталь.

Це фундаментальна межа обчислень.

Комп’ютер має лише:
- час
- памʼять

Будь-яке рішення — це перерозподіл між ними.

## 15. Що потрібно засвоїти

- Оптимізація часу часто коштує памʼяті.
- Памʼять може замінювати повторні обчислення.
- Іноді існує просто кращий алгоритм без компромісу.
- Контекст визначає, що важливіше.
